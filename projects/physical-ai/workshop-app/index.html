Physical AI Data Review Console

This is a walkthrough of the Workshop app I rebuilt for my Physical AI portfolio case. The app models how a team moves from collected robot episodes to staged review decisions, expert escalation, and preparation for model training. It uses representative demo data.

The premise is that collected episodes are not automatically training-ready data. The app’s job is to show where each episode breaks down, who should judge it, and what happens after the decision.

Dashboard: operating state before drilldown

The dashboard starts with operating state rather than raw tables. It tells the reviewer whether the current batch is moving toward a training-ready dataset, which segment cases required expert judgment, which episodes should be opened for deeper review, and which defect categories are reducing usable Physical AI data.

This chart turns reviewed Physical AI data into an operating signal. Instead of showing failed episodes as isolated cases, it groups defect attribution records by defect category, so the team can see whether usable training data is being lost to network issues, collector errors, platform/sensor issues, collection-method context, or sim-to-real gaps — giving Data PMs and Head Researchers a concrete starting point for follow-up action after segment review is closed.

Two ways into a decision

PATH 1
HR Escalation Lane
Dashboard → A. Head Researcher Escalation Cases → Segment Review Detail. The fast path for cases already routed to expert judgment.
PATH 2
Episode Drilldown
Dashboard → B. All Episodes → Episode Review Detail → Segment Review Detail. The full-context path for inspecting a collection run end to end.

The repeated design choice is to keep each surface at the right grain: batch state at the dashboard, collection context at the episode level, training-use judgment at the segment level, and operational correction at the action level.

Path 1 · Head Researcher escalation lane

This is the expert-judgment shortcut. From the Dashboard, the Head Researcher opens A. Head Researcher Escalation Cases and clicks directly into the segment that needs their judgment — skipping the full episode context, because the case has already been routed to them with a reason attached.

Dashboard → A. Head Researcher Escalation Cases → click selected segment (e.g. SEG_018_04) → Segment Review Detail — HR Escalation Lane
Step 1 · A. Head Researcher Escalation Cases

Review Data PM escalations that require Head Researcher judgment

grain: one row = one Subtask Segment escalated from Data PM to Head Researcher

This is not a generic error list. It isolates segment-level cases that the Data PM could not safely close alone. Each row preserves the episode, selected segment, subtask goal, Data PM decision, escalation reason, escalation timestamp, final training decision, and review decision state — so the Head Researcher does not need to restart from zero or inspect every episode. They receive the cases where judgment is ambiguous or policy-sensitive, with the prior routing reason still attached.

Step 2 · Segment Review Detail (HR Escalation Lane)

Clicking a row opens straight into the segment-level decision surface, with a fixed top context bar showing the selected segment and its parent episode, and a Segment Window / Open Video button.

  • Selected Segment Summary & Subtask Criteria
  • Segment Decisions in the Same Episode
  • Decision Evidence (video, sensor, raw, metadata)
  • Initial Reviewer Decision — read-only
  • Data PM Decision — read-only
  • Head Researcher’s Final Closure
  • Follow-up Action (links to Follow-up Action Detail)
  • Dataset Package Readiness
Step 3 · Follow-up Action Detail (if a defect is diagnosed)

If the closure identifies a defect, the reviewer can click through to a recommended action: target, owner, priority, due date, status, source defect, affected segment, and supporting evidence.

Path 2 · Episode review & drilldown

This is the full-context path. From the Dashboard, a reviewer opens B. Physical AI Data Collection Review — All Episodes, selects one episode (especially a failed or partially salvaged run), and drills from full run context down into a specific segment.

Dashboard → B. All Episodes → click selected episode (EP_018) → Episode Review Detail → click segment (SEG_018_04) → Segment Review Detail
Step 1 · B. All-episodes review table

Open any episode — especially failed or partially salvaged runs

grain: one row = one Collection Episode

The all-episodes table keeps decisions grounded in the original collection run. A reviewer can start from episode status, primary issue, source type, task, scenario, duration, and collection time before drilling into segment-level decisions. This prevents episode-level status from being confused with final segment-level training usability — a partial_salvage episode can still contain segments that are usable for training.

Step 2 · Episode Review Detail

Opening EP_018 shows full-run context before any segment-level decision is made — a fixed top context bar with an Open Episode Video button, followed by:

The Episode Lineage Graph is not a reviewer-facing primary screen. It is a traceability template that shows every ontology object connected to the selected Collection Episode: which session it came from, what export-readiness items exist, which QC signals fired, how those signals routed to segment review, and whether a defect attribution led to a follow-up action. Step 3 · Segment Review Detail (episode path)

Clicking SEG_018_04 inside the episode opens the same Segment Review Detail surface used in Path 1, scoped to this episode/segment pair.

What both paths share

Segment Review Detail, Follow-up Action Detail, and the video/media drawer are common reusable surfaces, not separate fixed pages. Whether a Head Researcher arrives through the escalation lane or through an episode drilldown, they land on the same kind of decision surface — the displayed evidence, staged decisions, and follow-up rows simply scope to whichever segment was selected.

That reuse is a deliberate design choice: it keeps the decision logic in one place instead of duplicating it per entry path, while still letting each path optimize for its own situation — speed for the escalation lane, full context for the episode drilldown.

For a Deployment Strategist role, this is the part I would want to discuss: the work is not only about building screens. It is about finding the real operating constraint, defining the right decision grain, and shaping a workflow that lets a customer move from messy evidence to action.
